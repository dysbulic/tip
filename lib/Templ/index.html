Templ Templating Engine and CMS

2007/04/17 — I'm currently MPP's web administrator. I get lots of requests for campaign sites that need to go up quickly. I'm very comfortable with HTML and CSS. I don't need or want a WYSIWYG editor. I write in emacs, and with things like regex matching and learn scripts, I work more quickly without the web prettiness. Once I've done a design though, I definitely don't want to be responsible for the myriad content edits that have to be subsequently done, nor do I expect anyone else to learn or enjoy the tools I use.

I know there are 800 different CMSes and probably 200 different templating systems. I've tried several of them and I don't really love any of them. I've thought about things I want and most of the popular CMSes are stronger on the CMS interface rather than the templating language. My interest is a system that allows me as a knowledgeable devloper to quickly produce sites. The later content edits by less technical people are important, but I want the system to be powerful first and then build tools that novice users need on top of that.

The Templ engine uses a two-stage generation process — a modular java-based system which compiles templates to HTML, SVG, PHP, etc. The reasons for this are varied:

  • For many sites there are processing operations on the data that are complex, but which only need to be done once. Having a part of the system that is run when content changes increases efficiency for certain dynamic sites.
  • Java servlets are memory intensive and servlets containers are not widely available on cheap hosting plans, so an all-java solution is not practical.
  • PHP is a fine language for small projects, but even with its increasing focus on object orieted design, it can be cumbersome for large projects.
  • Java virtual machines and PHP are common on hosting plans, so java's power can be leveraged without requiring a servlet container.

The basic idea is that a layout template is shared across a site. The different areas of the page come from an environmental context. This context is provided from a variety of sources. Currently for each output page there is an associated file that contains environment information (title, content, etc.)

HooksProcessor starts at some base directory and iterates over all the files in that directory running a set of transforms on them to produce the output documents. The default settings are discussed in another document.
